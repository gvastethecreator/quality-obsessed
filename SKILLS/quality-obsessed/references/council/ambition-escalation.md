# Ambition Escalation

Use this when the user asks for exceptional quality, says scope can expand, invokes `quality-obsessed` for a superior result, or when the requested artifact is likely to become forgettable if built literally.

The point is not scope creep. The point is to identify the stronger outcome hiding behind the literal request and build toward it with judgment, proof, and cuts.

## Contract

Before planning or execution, write:

```text
Literal request:
Underlying user outcome:
Competent literal version:
Superior version:
Ambition leap:
Why the leap serves the outcome:
Added scope:
Cuts:
Risk/cost:
Proof the leap wins:
Fallback if the leap fails:
```

## Escalation Ladder

Pick the highest rung that still fits user intent, repo constraints, available tools, and proof budget.

1. Better literal: same artifact, stronger craft and proof.
2. Expanded depth: same artifact plus richer states, recovery, detail, and inspection.
3. Stronger product: reframe the artifact around the user's outcome, not only the named object.
4. Signature leap: add one memorable capability, interaction, visual direction, system depth, or decision surface a competent pass would not attempt.
5. Showcase result: a cohesive artifact with strong direction, deep behavior, proof, and cuts; it should feel like the user got more than they asked for without losing the thread.

## Good Expansion

Expansion is accepted when all are true:

- it directly improves the user's main outcome.
- it is visible, usable, testable, or decision-improving in the artifact.
- it can be proved in the current environment.
- it does not hide a broken core path.
- Ponytail can still name what was cut.
- the final result is simpler to understand or more valuable to use, not merely larger.

## Bad Expansion

Reject expansion when it:

- adds features before the core path works.
- creates dependencies, APIs, assets, or data work the user did not need.
- makes verification vague or impossible.
- turns a focused prototype into a product shell with shallow panels.
- improves docs/process while the artifact stays ordinary.
- makes the result harder to use, maintain, or resume.
- violates explicit user limits, security boundaries, budget, release constraints, or repo conventions.

## Decision Rule

Default to the superior version when the user invites scope expansion or asks for standout quality. Ask only when the escalation changes product direction, durable architecture, spending, external dependencies, data/security posture, brand identity, or irreversible scope.

If the leap fails in proof, fall back to the strongest lower rung and record why. Do not sell failed ambition as quality.

## Completion

Ambition escalation is complete when:

- the literal request still works or is deliberately reframed with a reason.
- at least one signature leap lands in the artifact itself.
- the added scope has proof.
- the cuts are explicit.
- the reality verdict explains why the superior version beats the competent literal version.
