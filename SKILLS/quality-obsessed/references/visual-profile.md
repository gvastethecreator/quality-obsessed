# Visual Profile

Use this profile for UI, product, prototypes, games, images, interaction craft, and other rendered artifacts.

## Target read

Inspect an actual screenshot, render, canvas, or supplied reference before judging. Record:

```text
Current read:
Target read:
Primary action or focal state:
Supported viewports/platforms:
Brand or design-system constraints:
States that actually exist in the touched flow:
Proof needed:
```

Do not impose originality when the goal is native, brand, or design-system conformance. When standout identity is part of the target, name the positive signature the user should notice rather than merely banning templates.

## Craft lenses

Inspect hierarchy, layout, density, alignment, contrast, typography, affordances, input feedback, motion, accessibility, responsiveness, recovery, and performance only where they affect the target experience.

Render the happy path plus applicable empty, loading, error, overflow, disabled, and recovery states. A state or viewport that the product does not support is `N/A` with a reason, not a fabricated requirement.

Use the five-second read only when first impression matters: what is this, what should the user do, and does the screen express the intended product identity or system language?

## Artifact realism

Unless the user explicitly requests a mock-only artifact, exercise the main path against the real implementation and representative data. Every visible control must work or be clearly labeled noninteractive; error states must explain the failure and a useful recovery path. Lorem ipsum, placeholder items, and placeholder images cannot prove product readiness or ship as final assets.

```text
artifact_realism: representative-functional-no-placeholders
```

## Proof

Compare like-for-like screenshots or recordings. At least one early visual change should be visible without explanatory prose. Verify supported narrow/dense layouts, keyboard/focus/touch behavior, and text contrast when those surfaces are in scope.

Generated visual horizons are guidance, not evidence. Accept one only when it is feasible in the current stack and translates into named implementation moves.
