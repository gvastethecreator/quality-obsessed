# Quality Council

Use this when the mission needs more than one quality lens: broad work, persistence mode, ambitious output, recovery, unclear product direction, visual quality, risky code, or any result that must exceed the literal request.

The council can be internal notes, delegated reviewers, or subagents. The point is pressure, not bureaucracy. A council pass counts only when it changes the artifact, backlog, proof, scope, or decision.

## Default Stance

When `quality-obsessed` is invoked, do not optimize for avoiding ceremony. Optimize for useful pressure. It is acceptable to run council, grilling, critique, and review even on small tasks when that pressure can expose a better artifact.

The council should ask: what would make this result meaningfully more useful, more memorable, safer, clearer, more resilient, or easier to trust than the literal request?

## Roles

Pick the smallest set that can expose different failure modes:

- **Mission Owner:** keeps the user outcome, baseline, scope, and final claim honest.
- **Ambition Lead:** finds the superior version hiding behind the literal request.
- **Relentless Critic:** names why the current artifact may still be ordinary, bland, or weak.
- **Adversary:** attacks assumptions, hostile inputs, edge cases, and recovery.
- **Craft Lead:** checks visual/product/game/docs/code craft for first-use quality.
- **Proof Lead:** checks whether evidence actually supports the quality claim.
- **Simplifier:** cuts additions that make the artifact larger but not better.
- **Orchestrator:** picks domain skills, delegated reviewers, proof tools, and references.

## Council Pass

```text
Mission:
Baseline:
Literal request:
Superior version:
Roles used:
Top findings:
Ambition leap:
Risks or cuts:
Proof required:
Decision:
Next artifact move:
```

Each finding needs location, failure, user impact, fix sketch, and proof needed. Generic advice is discarded.

## When To Run

- before Loop 1 on persistence or ambitious missions.
- after Loop 1 if the foundation lift is not obviously stronger.
- after two `flat`, `mixed`, or `worse` verdicts.
- before Loop 30 continuation verdict.
- whenever the agent is about to stop mainly because the work feels "good enough".

## Hard Rules

- No council theater: a council pass must create or change a backlog item, artifact decision, proof target, or cut.
- No comfort consensus: if the artifact is weak, the council says red.
- No scope expansion without user-outcome fit, proof, cuts, and fallback.
- No tiny-task exemption when the user explicitly invoked obsession; small tasks can still deserve council pressure.
- No hidden decisions: accepted and rejected council findings must be recorded in the task record or final response.
